# readme_for_ai

## 目录职责

这个目录放自动化测试，主要负责验证底层逻辑是否仍然成立。

当前测试覆盖很浅，主要是为了固定最小可运行链路中的两个稳定点：
- 日志行解析
- 规则判定

## 在整体项目中的位置

`tests/` 当前主要验证 `src/rocoworld_yise_detector/` 中的低层模块。

它还没有覆盖：
- 配置加载层
- watch 模式日志读取层
- 多文件样本回放

因此如果未来改动这些部分，通常要补测试，而不是只改代码。

## 关键路径

### `tests/test_parser.py`

- 验证 `EncounterParser` 是否能提取标准字段，并忽略非遭遇行。
- 改字段提取规则、改 `line_filters` 行为、改标准字段结构时优先看这个测试。

### `tests/test_detector.py`

- 验证 `ShinyDetector` 是否能区分命中规则和未知规则。
- 改匹配优先级、改 `MonsterVariant` 结构、改 `UNKNOWN` 逻辑时优先看这个测试。

### `tests/test_cli.py`

- 验证 CLI 在日志文件不存在时是否给出深度日志前置条件提示。
- 改 `scan` / `watch` 的缺失日志提示、命令入口编排时优先看这个测试。

## 当前测试组织方式

- 当前测试通过在 `sys.path` 中插入 `src/` 直接导入源码。
- 没有使用 `pytest` 特性，仍然是 `unittest` 风格。
- 当前样例偏“单文件单场景”，还不是完整回归测试集。

## 修改落点指引

### 如果要新增解析字段

优先补：
- `tests/test_parser.py`
- 若字段参与判定，再补 `tests/test_detector.py`

### 如果要修改判定策略

优先补：
- `tests/test_detector.py`
- 若涉及真实日志到结构化字段的联动，再考虑新增更高层的集成测试

### 如果要重构 watcher 或 CLI

当前对 watcher 仍没有现成覆盖；CLI 缺失日志提示已有基础覆盖。重构相关逻辑时，通常应补测试，而不是只依赖手工验证。

建议新增方向：
- 配置加载测试
- 样本文件扫描测试
- watch 模式下新增行、日志截断、编码切换测试

## 边界与约定

- 测试应优先固定可观察行为，而不是复制实现细节。
- 如果未来引入真实日志脱敏样本，优先从 `samples/` 引用 fixture，不要在测试文件里硬编码大量日志文本。
- 当规则库结构发生变化时，要同步更新测试，使失败能直接提示是 schema 变了还是行为变了。

## 当前待确认

- 未来是否继续保持 `unittest`，还是迁移到 `pytest`，当前还没必要下结论。
- 真实日志到手后，测试是否需要按“解析层 / 判定层 / 集成回放层”重组，目前尚未确定。
